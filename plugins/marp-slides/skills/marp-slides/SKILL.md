# Marp Slides

Create HTML presentations from a directory of markdown slide files.

**Requires**: [Marp CLI](https://github.com/marp-team/marp-cli) — install via `npm install -g @marp-team/marp-cli`

## Workflow

1. **Setup**: Each presentation lives in its own directory
2. **Author**: Create individual `.md` files for each slide (sorted lexically)
3. **Style**: Customize `style.yaml` for deck appearance (optional)
4. **Build**: Run `/marp-slides:build` to combine and render
   - Or separately: `/marp-slides:combine` then `/marp-slides:render`
5. **Review**: Open `slides.html` and verify slides render correctly

## Commands

| Command | Purpose |
|---------|---------|
| `/marp-slides:build [path]` | Full build: combine + render (default: cwd) |
| `/marp-slides:combine [path]` | Merge slide files into `combined_slides.md` |
| `/marp-slides:render [path]` | Render `combined_slides.md` to `slides.html` |

## File Structure

```
presentation/
├── 01_title.md
├── 02_intro.md
├── 03_main_point.md
├── ...
├── style.yaml          # deck styling (created automatically if missing)
├── combined_slides.md  # generated by combine
└── slides.html         # final output
```

**Naming**: Files combine in lexical sort order. Use numeric prefixes (`01_`, `02_`) or any scheme that sorts correctly.

**Individual slide files**: Raw markdown content only. No frontmatter, no `---` separators. Start each file with an H1 title.

## Style Customization

Deck styling lives in `style.yaml` in the slide directory. To customize appearance:

1. Edit `style.yaml` directly
2. Re-run `/marp-slides:build`

**Default style structure:**

| Element | Default | Purpose |
|---------|---------|---------|
| `section` | 22px font, 40px padding | Base text size and margins |
| `section h1` | Absolute top position, 36px | Title pinned at top of slide |
| `table` | 18px font, 100% width | Compact tables |
| `pre`, `code` | 15px font | Compact code blocks |
| `.fit` class | 85% scale | Shrinks dense content to fit |

The H1 is absolutely positioned at top (30px from top edge). Size images to avoid collision with this zone.

## Marp Directives

Insert directives in individual slide files for per-slide control:

| Directive | Effect |
|-----------|--------|
| `<!-- _class: fit -->` | Scale down content (auto-added for dense slides) |
| `<!-- _class: invert -->` | Dark background, light text |
| `<!-- _class: lead -->` | Centered, larger text |
| `<!-- _backgroundColor: #123 -->` | Custom slide background |
| `<!-- _color: #fff -->` | Custom text color |

The combine script auto-adds `<!-- _class: fit -->` when a slide has 2+ code blocks or 8+ table cells.

## Image Handling

```markdown
![height:340px](image.png)
```

- Store images in the slide directory
- Use height constraints to fit below the H1 zone (340-400px typical)
- If an image has its own title, remove it to avoid redundancy with slide H1

## Slide Style Guide

**Structure**
- One idea per slide — if you need "and", consider splitting
- Title states the *point*, not just topic ("KV Cache Reduces Compute" vs "KV Cache")
- Build complexity gradually — introduce concepts before details

**Content**
- Short phrases over full sentences
- Parallel structure in bullet lists
- Bold key terms, italics for sources/footnotes
- If a bullet needs sub-bullets, consider splitting the slide

**Code & Tables**
- Show only relevant parts — minimize boilerplate
- ASCII diagrams are often clearer than complex graphics
- Tables work best for comparisons (2-4 columns)
- Bold key data points in tables

**DRY Principle**
- Each slide should add new information
- If slides repeat content, consolidate or restructure
- Review the deck for redundancy before finalizing

## Quality Review

After rendering, open `slides.html` and check:

- [ ] Text is readable, not overflowing
- [ ] Images fit below headers, no collision
- [ ] Code blocks are legible
- [ ] Tables don't extend beyond slide
- [ ] Consistent styling across slides
- [ ] No redundant information between slides
